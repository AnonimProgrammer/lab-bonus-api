# Notes: my backend design log

**Live URL (Vercel):** https://lab-bonus-api.vercel.app

## 1. The GET handler

- In the App Router, the folder path under `app/` becomes the URL. A file named
  `route.js` inside `app/api/users/` is a Route Handler for that path, so
  `app/api/users/route.js` answers `/api/users`. Exporting a function called
  `GET` tells Next.js to run it on GET requests to that address.
- I opened `http://localhost:3000/api/users` directly in the browser first. When
  I saw the three seeded users as raw JSON, I knew the handler worked before
  checking the main page.

## 2. The POST handler

- The request body arrives as a stream of bytes over the network, not a plain
  JavaScript object. `await request.json()` reads that stream and parses it into
  JSON. You cannot read it synchronously because the data may still be arriving
  when your function starts.
- The `id` is generated by Supabase when the row is inserted (a UUID default on
  the column). Each user needs a unique `id` so React can key list items, DELETE
  can target one row, and two users with the same name do not collide.
- I return **201 Created**. That status means "something new was successfully
  created," which is more precise than 200 for a POST that adds a resource.

## 3. The shared array

- When I first used `app/data/users.js`, both `GET` and `POST` imported the same
  exported array. Node loads that module once and caches it, so both handlers
  read and write the exact same in-memory array — not separate copies.
- After a dev server restart, added users disappeared because the data only lived
  in the running process's RAM. Stopping the server wiped that memory, and the
  module loaded fresh with just the three seeded entries again. Nothing was
  written to disk or a database until we added Supabase.

## 4. Validation

- Before saving, `validateUserInput` trims the fields and checks that `name`,
  `imageUrl`, and `description` are all present, and that `imageUrl` is a real
  URL. If anything fails, the handler returns **400** with a JSON body like
  `{ "error": "..." }`.
- The form checks `res.ok` and shows a red error under the button, e.g. "The
  server said no (status 400)." The user sees that something was rejected without
  a broken card being added.

## 5. The persistence question (and the bonus)

- On Vercel with the in-memory array, users I added could vanish after a
  redeploy or cold start. Each serverless instance had its own copy of the array
  in RAM, so there was no shared storage between instances or restarts.
- With Supabase wired up, the Route Handlers now call `getUsers`, `createUser`,
  and `deleteUser` in `app/lib/users.js` instead of pushing to the array. The
  frontend did not change at all — it still fetches `/api/users` the same way.
  That is the point of a clean API boundary: swap the storage behind the URL and
  the UI keeps working.
- One thing that tripped me up: I accidentally put the **anon** key in
  `SUPABASE_SERVICE_ROLE_KEY`, which caused RLS to block inserts (error 42501).
  Switching to the real service role key fixed it.
